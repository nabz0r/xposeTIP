### 🔴 NEW BUGS DISCOVERED THIS RUN

1. **varchar(500) overflow in `scraper_engine`** (S1 repro). `psycopg2.errors.StringDataRightTruncation: value too long for type character varying(500)` from `api/tasks/module_tasks.py:165`. One scraper produced a value exceeding 500 chars and the existing pre-flush truncation in `module_tasks.py` did not cover the column that overflowed. Cost: the ENTIRE scraper_engine module failed and 116 scrapers' worth of findings were dropped for that scan. Repro: re-run a scan on `nabz0r@gmail.com`. Action: find which `findings.*` column is `VARCHAR(500)` (candidates: `title`, `value`, `url`, `description`, `evidence_url`) and either widen to `TEXT` or extend truncation to cover it.

2. **`_auto_resolved_name` is null for both targets despite scraper data being present.** T1 has a GitHub avatar from a profile that almost certainly carried a `name` field; T2's persona spans 25 platforms. The profile aggregator's name-resolution path is failing silently. Downstream impact: every first_name/name-input scraper is permanently blocked (this is upstream of the S122b name-input dispatch gap — fixing only the dispatch won't help if no name reaches it).

3. **L4 analyzer attribution is collapsed under module=`intelligence`**. Cannot tell from the DB which of the 9 analyzers ran on a given scan. Either label findings with the analyzer that produced them, or persist a per-analyzer status alongside `module_progress`. Otherwise D.5 audits are blind.

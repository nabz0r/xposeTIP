### S152 anomaly — Cancel race vs chord sub-tasks
After Cancel set status=`cancelled` at 14:56:23, the orchestrator's chord sub-tasks (already dispatched module tasks) continued running. At 14:57:10, the chord's `finalize_scan` callback fired and **overwrote** `status=cancelled → status=completed`. Subsequently, the cascade entered `gathering` and got stuck there because finalize_scan ran in an inconsistent state.

End state for T3 right after cancel: `status=completed cascade=gathering` — and `/system/scans/live` **picked it up again** because of the wider predicate (status=completed AND cascade non-terminal). T3 reappeared in Live Scans.

Manual cleanup: set `cascade_state='failed'` on T3 to escape the stuck state. Watchdog would NOT have caught this since it only checks status in (`queued`, `running`), not `completed`.

**Root cause**: `superadmin_cancel_scan` revokes only the parent `launch_scan` task (via `celery_task_id`), not the chord of module sub-tasks. Module tasks complete normally → chord callback fires `finalize_scan` → status flipped back from `cancelled` to `completed`.

**Fix idea (out of scope for this smoke)**: cancel should ALSO revoke each `module_task` from the dispatched chord, OR set a sentinel field on Scan that finalize_scan checks before any mutation (e.g., "if status=='cancelled' on entry, return early").

## Anomalies / regressions discovered

1. **S152 Cancel race** (NEW): `superadmin_cancel_scan` revokes only the parent `launch_scan` task. Already-dispatched chord sub-tasks complete normally and `finalize_scan` overwrites `status=cancelled` back to `completed`. End state: `status=completed cascade=gathering` (stuck). Reappears in Live Scans. Watchdog won't catch (only checks queued/running). Fix idea: revoke all module sub-tasks AND check status='cancelled' at top of finalize_scan to return early. **Priority: P2** (cancel works visually but doesn't actually stop the work).

2. **`/system/stats` top-level numbers appear global, not workspace-scoped** (216 scans, 751 findings identical in Default and Ferrero workspaces). Recent_activity section IS workspace-scoped. This is unrelated to S152 but worth noting — operators using the dashboard for per-workspace counts will be confused.

3. **Smoke-script bug (not product)**: My direct `launch_scan.delay()` after `session.flush()` (without commit) raced PostgreSQL READ COMMITTED. Real API path commits before dispatch and avoids this. T3 had to be re-dispatched. Documenting so future smokes don't fall in the same hole.

## Recommendation

**Ship as-is for S148, S149, S150, S152 part B (workspace refresh).** All three are doing what they say on the tin.

**S152 part A (Cancel button) needs a follow-up sprint**: the cancel race against chord sub-tasks is a real bug. The Cancel button gives operators false confidence — they click, see the scan disappear, but the work continues server-side until finalize_scan races back. Suggest S153 to either:
- Revoke all dispatched module task IDs (requires storing them on Scan)
- OR add an early-return check in finalize_scan that no-ops if `status='cancelled'`

The S148 watchdog DOES NOT catch this case (only checks queued/running), so a stuck `status=completed cascade=gathering` row sits in Live Scans indefinitely. Cleanup currently requires manual SQL.

**S150 deserves a targeted synthetic test** (not full smoke) to prove the get-or-create branch executes correctly. A 10-line Python script that constructs the scenario directly is more reliable than waiting for scrapers to re-discover a duplicate URL.

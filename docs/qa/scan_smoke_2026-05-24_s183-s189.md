- **Workspace auto-reverted between scans 2 → 3.** After leaving the Crypto-workspace target detail page for ~10 min while waiting for scan 2 to complete, navigating to scan 3's UUID landed me back in the `Ferrero` workspace (the user's original default at login). Recovered by clicking the workspace switcher and re-selecting Crypto. **Likely behavior:** the JWT carries the workspace at issue time, and somewhere between page loads the cookie/local-storage state lost the explicit switch. Worth a separate dig — this could affect any user doing a long-running multi-tab investigation.
- **Aleksei early-profile name was "Aleksei Talisainen" but final stored as "Aleksei Mustafin".** PASS2 fired against the EARLY name (Talisainen), not the final. If the search corpus for PASS2 ever shifts, this would matter. The DB-stored primary_name reflects post-PASS2 re-aggregation. Not a bug, but worth noting as a per-scan timing artifact: PASS2 search queries are bootstrapped from the A3 early profile, not the final post-aggregation result.
- **S187 SEC EDGAR / Companies House did not appear in PASS2 fan-out for any of the 3.** This is the most concrete unexpected finding. Need a focused investigation of the `name_scraper_engine` dispatch table to confirm whether S187 scrapers are reachable on a name-only input or whether they require an upstream `legal_record`-typed seed.
- **`behavioral_hashes_computed` did NOT increment despite 3 successful scans.** All 3 targets had a prior `bfp_behavioral_hash_v1` from previous scans (each had been scanned before — e.g., target 1's overview showed prior Fingerprint Evolution history). The recompute path correctly OVERWRITES the existing hash rather than appending. Count stable = correct, but worth documenting.
